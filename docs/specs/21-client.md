# 21 — Клиент (The Client): архитектура R3F-приложения, сетевой слой, перф-бюджеты, заглушки-ассеты

> **Статус:** черновик спеки v0.1 · зависит от `docs/specs/00-canon.md` (канон v1.0).
> **Скоуп файла:** техническая архитектура нового клиента `sunnyside/` — Vite + React + React-Three-Fiber (R3F) + zustand; структура папок; сцены (ферма/город/ярмарка/смена); реестр zustand-слайсов и их стейт; сетевой слой на `supabase-js` (оптимистичные апдейты, reconnect, оффлайн-очередь мутаций); тик-синхронизация серверных таймеров; перф-бюджеты (полигоны, draw calls, 60 fps на средних ноутбуках); система заглушек-ассетов (примитивы + палитра, конвенция имён, реестр `placeholder→final`); дебаг-роутинг (`?screen=`); тестовая стратегия (vitest — юниты экономики, Playwright — смоук).
> **Не в скоупе:** серверная схема БД, RLS, Edge Functions и их контракты (это спека бэкенда, `docs/specs/20-backend.md` — нейминг-кандидат, требует создания); дизайн экранов и их визуал (`docs/specs/19-ui-ux.md` — нейминг-кандидат); арт-пайплайн финальных GLB (`docs/specs/18-art.md` — нейминг-кандидат); игровые правила систем (их профильные спеки §6). Здесь — **как клиент устроен и как он говорит с сервером**, а не что именно он показывает.
> **Правило языка (канон §5):** русский текст; нейминги — `English (Русская локализация)` при первом упоминании; в коде/БД — только английский snake_case ключ.
> **Правило соседства с прототипом:** существующий корневой прототип `src/` (проект `farm-truck`) **не трогаем**. Новый клиент живёт в изолированной подпапке `sunnyside/` с собственным `package.json` и `vite.config.ts`. Причина изоляции — §3.1.

## 1. TL;DR

- **Отдельное приложение `sunnyside/`.** Клиент Sunnyside — новый Vite-проект в подпапке репозитория, со своим `package.json`, `vite.config.ts`, `tsconfig`. Корневой прототип `farm-truck` (`src/`) остаётся нетронутым как референс-песочница; код между ними **не шарится импортами** (граница по файловой системе, не по alias).
- **Стек фиксирован каноном (D13/D14):** Vite 5 + React 18 + `@react-three/fiber` 8 + `@react-three/drei` 9 + zustand 5, пакетный менеджер **pnpm**. Бэкенд — Supabase (`@supabase/supabase-js` v2): Postgres + Realtime + Edge Functions. Мультиплеер **асинхронный** — реалтайм-физики нет, только подписки на изменения строк.
- **Жёсткая граница `game/ ↔ scene/`** наследуется от прототипа (CLAUDE.md): вся игровая логика и сторы — ноль импортов из `three`/`@react-three/*`, тестируются в node. `scene/` — только рендер. Экономические формулы (`14-economy`) живут в `game/econ/` как чистые функции и покрываются vitest.
- **Четыре сцены-корня, один `<Canvas>` на активную сцену:** `FarmScene (Ферма)`, `TownScene (Город)`, `FairScene (Ярмарка)`, `ShiftScene (Смена у прилавка)`. Переключение — не роутером URL в проде, а `sceneSlice.active`; `?screen=` — дебаг-оверрайд для разработки и Playwright.
- **12+ zustand-слайсов** в одном сторе (`useStore`), скомпонованном из slice-функций: `session`, `clock`, `net`, `farm`, `inventory`, `econ`, `demand`, `coop`, `fair`, `event`, `town`, `progression`, `collections`, `ui`, `scene`. Персист — только клиентский UI/кэш; истина по экономике — сервер.
- **Оптимистичный сетевой слой.** Каждая мутация игрока применяется локально мгновенно, ставится в **оффлайн-устойчивую очередь** (`mutationQueue`, persisted в IndexedDB), шлётся на Edge Function; при подтверждении — reconcile, при конфликте — сервер-побеждает + мягкий откат с тостом. Reconnect с экспоненциальным backoff, при возврате онлайна очередь дренажится по порядку.
- **Тик-синхронизация с сервером.** Клиент **не доверяет** `Date.now()` для игровых дедлайнов: держит `serverOffset` (разница серверных и клиентских часов, замеряется при коннекте), все таймеры (рост грядки, готовка, окно ярмарки, rollover) считаются от `serverNow() = Date.now() + serverOffset`. Завершение таймера **не начисляет награду локально** — только показывает «готово», забор ресурса — отдельная подтверждаемая мутация.
- **Перф-бюджет: 60 fps на среднем ноутбуке 2020 г. (Intel Iris / M1, 1080p).** Потолки: ≤ 150 draw calls/кадр, ≤ 350k треугольников в кадре, ≤ 12 уникальных материалов, инстансинг для всех повторяющихся пропсов, один `AnimationMixer` на тип. Город (100–200 ферм) рисуется имперостерами/LOD, не полными мешами.
- **Заглушки-ассеты — примитивы + палитра.** До прихода финальных GLB всё рисуется box/cylinder/cone из общей палитры (`palette.json`, наследует эстетику прототипа: `MeshLambertMaterial`, плоские цвета, ноль PBR). Реестр `assetRegistry` мапит `placeholder_key → final_glb_path`; замена ассета = правка одной строки реестра, не компонента.
- **Тесты:** vitest — юниты экономических формул и slice-редьюсеров (обязателен ≥ 90% покрытия `game/econ/`); Playwright — смоук-прогоны ключевых экранов через `?screen=` с мок-сервером (`msw`-подобный перехват Supabase). CI-гейт: `pnpm test && pnpm build` зелёные до мержа.

## 2. Player Experience (как это ощущается)

Клиент — невидимая система: игрок не должен её замечать. Но три её свойства он чувствует напрямую.

### 2.1 Мгновенность (оптимистичность)
Игрок кликает по грядке — семя падает **сразу**, без спиннера. Кликает «полить» — лейка плеснула в тот же кадр. Сеть работает в фоне: подтверждение сервера приходит через 100–400 мс и игрок его не видит, потому что UI уже показал результат. Ощущение — «игра отзывчивая, как одиночная», хотя всё проходит через сервер (пиллар P3: игра дружелюбна, не мучает лоадерами).

### 2.2 Прощение к сети (оффлайн и обрыв)
Метро, лифт, слетел Wi-Fi — игрок продолжает сажать, поливать, готовить. Сверху появляется мягкая плашка `Working offline — we'll catch up (Оффлайн — синхронизируем позже)` в тёплом тоне, без красного алерта. Когда связь вернулась — плашка сменяется на `All caught up! (Всё синхронизировано!)` и очередь тихо дренажится. Ничего не теряется, ничего не «сгорает» (канон E6, пиллар P3). Единственное, чего нельзя офлайн — увидеть **свежие** данные города/ярмарки (они серверные), но своя ферма полностью играбельна.

### 2.3 Честность таймеров
«Ярмарка закроется через 4:12:30» — этот таймер идёт от **серверного** времени, а не от часов ноутбука. Игрок перевёл часы вперёд — ничего не ускорилось (готовка не читерится клиентом). Таймер закончился, пока игра была закрыта — при возврате блюдо уже «готово к сбору», а не «начинает готовиться». Ощущение — мир живёт по общему серверному пульсу (пиллар P2), одинаковому для всего стрита.

### 2.4 Гладкость картинки
На среднем рабочем ноутбуке 3D-ферма крутится плавно: клик по слоту отзывается без лага, камера едет без рывков, город из сотни ферм не роняет fps. Даже на слабой машине включается `Lite mode (Лёгкий режим)` — меньше травы, проще тени — но игра остаётся той же. Никто не выпадает из-за железа (пиллар P3, инклюзивность).

## 3. Механики (техническая архитектура — исчерпывающе)

### 3.1 Изоляция приложения `sunnyside/`

**Почему подпапка, а не переезд в корень.** Корневой прототип `farm-truck` (`src/`, `package.json` с `name: "farm-truck"`) — рабочий 2D→3D эксперимент, привязанный к `reference/farm-truck-game.html` и Blender-пайплайну в `tools/`. Он остаётся источником проверенных паттернов (граница `game/scene`, инстансинг, persist-стор). Sunnyside — полноценный продукт с сетью, поэтому получает **свой** корень сборки, чтобы:
1. независимые версии зависимостей (Sunnyside тянет `@supabase/supabase-js`, `idb`, `@playwright/test` — прототипу они не нужны);
2. отдельный Vite/TS-конфиг (свои env, alias, worker-и) без риска сломать прототип;
3. чистый `pnpm --filter` таргет и отдельный деплой.

**Структура репозитория после добавления клиента:**

```

Нарушение границы — баг ревью, не «прагматичное решение».

### 3.2 Точка входа и провайдеры

`main.tsx` монтирует `<App/>` без React-Router (см. §3.8 — роутинг сцен внутри стора). Порядок инициализации при старте:
1. Прочитать `?screen=`, `?seed=`, `?town=` из `location.search` → `ui.debug`.
2. Создать Supabase-клиент (singleton, `net/client.ts`), поднять сессию (анонимная или по magic-link, спека бэкенда).
3. Замерить `serverOffset` (§3.6) первым RPC `get_server_time`.
4. Гидрировать persisted-слайсы из IndexedDB (незасинканная очередь + UI-кэш).
5. Подписаться на Realtime-каналы своего стрита/города (`net/channels.ts`).
6. Дренажить оффлайн-очередь, если она непуста.
7. Смонтировать активную сцену (`scene.active`, по умолчанию `farm`).

`App.tsx` — свитч по `scene.active`: рендерит ровно **один** `<Canvas>` для активной 3D-сцены + DOM-`<HUD/>` поверх. Переход между сценами размонтирует старый `<Canvas>` (освобождение GPU-памяти) и монтирует новый; общий кэш геометрий/материалов держится в модульном синглтоне `scene/assets`, чтобы не пересоздавать примитивы.

### 3.3 Сцены (4 корня)

Каждая сцена — самодостаточный R3F-граф. Общие узлы (камера, свет, `Ground`, `<PerfGate/>` из §3.9) живут в `scene/common/` и переиспользуются.

| Сцена | Ключ | Что показывает | Камера | Источник данных | Интеракция |
|---|---|---|---|---|---|
| **Farm Scene (Ферма)** | `farm` | Личная ферма: грядки, животные, станки, постройки (канон §3.8), дайнер | Орбитальная, полу-топ-даун, ограниченный зум | свои слайсы `farm`/`inventory` (локально-оптимистично) | клик по слоту/станку/зданию → экшен |
| **Town Scene (Город)** | `town` | Ярмарочная площадь + улицы (10–20 стритов × фермы), town projects (канон §3.7) | Облёт площади, клик по стриту → фокус | серверные снапшоты `town` (Realtime) | клик по ферме соседа → визит; по проекту → вклад |
| **Fair Scene (Ярмарка)** | `fair` | Прилавок игрока + ряды прилавков стрита; конкурсы, Appetite Meter | Ближе к прилавку | `fair`/`event` (Realtime, окно 36 ч) | выкладка стока, цена, голос в конкурсе |
| **Shift Scene (Смена у прилавка)** | `shift` | Активная мини-игра подачи: очередь гостей, поднос, сборка блюда | Фиксированная, «от прилавка» | локальный `fair.shift` (соло-смена, канон D4) | сборка подноса кликами, подача |

**Правила сцен:**
- Только `farm` и `shift` полностью играбельны офлайн (свои данные). `town` и `fair` требуют свежих серверных данных — офлайн показывают последний кэш-снапшот с плашкой `Snapshot — reconnecting (Снимок — переподключаемся)`.
- Сцена `shift` — таймер-мини-игра (наследует паттерн `TruckState` прототипа: очередь, patience, spawn), но тики считаются локально в пределах открытой смены; итог смены — **одна** подтверждаемая мутация (сумма чаевых/продаж), не по-клиентному начисление за каждого гостя (анти-чит, §3.6).
- Переход сцен инициируется UI (кнопки «В город», «На ярмарку») или календарём (наступила суббота → доступна кнопка `fair`). Сам переход = `scene.goto('fair')`, размонтирование/монтирование `<Canvas>`.

### 3.4 Стор: реестр zustand-слайсов

Один корневой `useStore = create<StoreState>()(persist(subscribeWithSelector((set,get)=>({ ...slices }))))`. Слайсы — фабрики `createXSlice(set,get)`, скомпонованные в `store/index.ts`. Персист (`partialize`) — **только** белый список полей (UI-настройки, кэш города, оффлайн-очередь-хэндл); игровая истина не персистится (она серверная, тянется при коннекте).

| Слайс | Ключ | Ключевой стейт (сокр.) | Персист? | Источник истины |
|---|---|---|---|---|
| Session | `session` | `userId, townId, streetId, farmId, authStatus` | нет | сервер |
| Clock | `clock` | `serverOffset, weekPhase (mon_plan…sun_event), rolloverAt, fairWindow{open,close}` | нет | сервер (`01-core-loop`) |
| Net | `net` | `online, reconnecting, queueLen, lastSyncAt, channelStatus` | частично (queueLen) | клиент |
| Farm | `farm` | `slots[], buildings{bld_*}, animals[], machines[]` | кэш | сервер, оптимистично локально |
| Inventory | `inventory` | `items: Record<productKey, qty>, storage{silo,icehouse}` | кэш | сервер |
| Econ | `econ` | `bucks, dimes, tickets, ribbons, farmValue` | нет | сервер (никогда не персистить баланс!) |
| Demand | `demand` | `board: Record<category, multiplier>, seed, nostalgia[]` | кэш | сервер (`14-economy`) |
| Coop | `coop` | `orders[], myContrib, deadline (thu 23:59)` | кэш | сервер (`01-core-loop`) |
| Fair | `fair` | `stall[], prices, shift?: ShiftState, contests[]` | нет | сервер + локальная смена |
| Event | `event` | `appetite (0..100), milestones[], myContribHist, tickets` | кэш | сервер (`10-server-event`) |
| Town | `town` | `streets[], projects{tp_*}, roster[]` | кэш | сервер (`11-town`) |
| Progression | `progression` | `farmLevel, knowHow{kh_*}, routePass{tier,track}` | кэш | сервер (`13-progression`) |
| Collections | `collections` | `toys{toy_*}, postcards[], ribbons[], cosmetics{cos_*}` | кэш | сервер (`17-collections`) |
| UI | `ui` | `debug, activePanel, toasts[], liteMode, locale` | да | клиент |
| Scene | `scene` | `active: 'farm'|'town'|'fair'|'shift', cameraTarget` | да (active) | клиент |

**Правила стора:**
- Экшены-мутации игрока идут через **единый враппер** `applyMutation(kind, payload)` (§3.5): он делает оптимистичный `set`, кладёт в очередь и возвращает промис подтверждения. Прямых `set` в сеть-зависимых слайсах из UI нет.
- `econ.bucks/dimes/tickets/ribbons` **никогда** не читаются из persist (защита от подмены баланса в localStorage) — только из серверного снапшота. Оптимистичная дельта показывается поверх серверного значения как `pendingDelta`, схлопывается при подтверждении.
- Селекторы — через `subscribeWithSelector` + `useShallow`, чтобы `scene/` перерендеривался точечно (перф, §3.9).

### 3.5 Сетевой слой (`net/`)

**Клиент.** `net/client.ts` — синглтон `createClient(url, publishableKey, { realtime:{ params:{ eventsPerSecond: 5 }}})`. Ключи — из `import.meta.env` (`.env.sunnyside`, уже в `.gitignore`). Секретный ключ в клиент **не** попадает — только `SUPABASE_PUBLISHABLE_KEY`.

**Три канала связи:**
1. **Мутации игрока → Edge Functions (RPC).** Все действия, меняющие серверный стейт (посадить, собрать, выложить на прилавок, вложить в котёл), идут `POST` на Edge Function, а не прямым `insert/update` (сервер валидирует правила и античит). `net/rpc.ts` — типизированные обёртки `rpc.plant({slotId,seed})` → `{ ok, serverState, error }`.
2. **Чтение снапшотов → REST (`.from().select()`).** Первичная гидрация слайсов при входе/смене сцены.
3. **Живые обновления → Realtime.** Подписка на изменения строк своего стрита/города (Appetite Meter, кооп-вклады соседей, town projects). `net/channels.ts` мапит `postgres_changes` → патч в слайс.

**Оптимистичная модель (сердце слоя).** Каждая мутация проходит конечный автомат:

```
optimistic → queued → in-flight → confirmed        (happy path)
                          └────→ conflict → rollback + toast
                          └────→ offline → (ждём online) → in-flight
```

- `applyMutation(kind, payload)`:
  1. Присвоить `clientMutationId = uuid()` (идемпотентность на сервере).
  2. Оптимистичный `set` в слайс + запомнить **обратный патч** (inverse) для отката.
  3. `queue.enqueue({id, kind, payload, baseVersion})` в IndexedDB (`net/queue.ts`, либа `idb`).
  4. Если online — сразу `flush()`; иначе ждать `online`.
- `flush()` дренажит очередь **строго по порядку** (FIFO, по одной — сохранение причинности: собрать нельзя раньше, чем посадить). На каждый элемент — RPC с `clientMutationId`. Ответ:
  - `ok`: применить авторитетный `serverState`, снять оптимистичную дельту, `queue.ack(id)`.
  - `conflict` (например, слот уже занят / недостаточно средств на сервере): применить обратный патч, показать тёплый тост `That one didn't take — try again (Не прошло — попробуй ещё)`, `queue.drop(id)`.
  - сетевая ошибка/5xx: оставить в очереди, backoff-ретрай.

**Оффлайн-очередь.** Персистится в IndexedDB (не localStorage — объём и структурированность). Переживает перезагрузку вкладки и полное закрытие. При старте (`main.tsx` шаг 6) недренированная очередь доигрывается. Идемпотентность — по `clientMutationId`: сервер хранит выполненные id (окно 24 ч), повтор возвращает закэшированный результат, не дублирует эффект.

**Reconnect.** `net.online` слушает `navigator.onLine` + heartbeat к Realtime. При обрыве:
- статус `reconnecting`, экспоненциальный backoff `min(30s, 2^n · 500ms)` с джиттером;
- Realtime-каналы пересоздаются при возврате, затем **делта-ресинк**: тянем снапшоты town/fair/event заново (они могли уйти вперёд), свои слайсы — reconcile с сервером.
- После успешного ресинка: замерить `serverOffset` заново (§3.6), дренажить очередь, тост `All caught up! (Всё синхронизировано!)`.

**Порядок при конфликте версий (сервер-побеждает).** Каждый серверный объект несёт `version` (bump на каждый write). Мутация шлёт `baseVersion`; если серверная > base — сервер решает по типу: аддитивные (вклад в котёл) применяются поверх; эксклюзивные (занять слот) отклоняются в `conflict`. Клиент никогда не «спорит» — принимает серверный стейт как истину (пиллар P3: без фрустрации, мягкий откат).

### 3.6 Тик-синхронизация таймеров с сервером

**Проблема:** игровые дедлайны (рост грядки, готовка станка, окно ярмарки 36 ч, rollover Вс 23:59, дедлайн кооп-заказов Чт 23:59 — канон §2.3) — **серверные**. Клиентский `Date.now()` может врать (сбитые часы, ручной перевод, дрейф).

**Решение — `serverOffset`:**
- При коннекте RPC `get_server_time` → `t_server`; `serverOffset = t_server − Date.now()` (усредняется по 3 замерам с поправкой на RTT/2, отбрасывая выбросы).
- `clock.serverNow() = Date.now() + serverOffset` — единственный источник времени для игровой логики. `Date.now()` напрямую в `game/` запрещён (lint-правило-кандидат).
- Пересчёт offset: при каждом reconnect и раз в 10 мин (дрейф часов).

**Модель таймеров — «дедлайн, не обратный отсчёт».** Клиент не тикает счётчики раз в секунду ради логики. Каждый таймер хранит **абсолютный** `readyAt` (серверное время готовности), пришедший с сервера. UI считает остаток как `readyAt − serverNow()` в `requestAnimationFrame`/интервале **только для отображения**. Логика «готово?» — булев `serverNow() ≥ readyAt`.

**Ключевое анти-чит-правило:** истечение таймера на клиенте **не начисляет** ресурс. Оно лишь меняет визуал на «готово к сбору». Забор — отдельная мутация `collect(slotId)`, сервер проверяет `now ≥ readyAt` по своим часам и только тогда выдаёт продукт. Перевод часов вперёд не ускоряет ничего.

**Смена (Shift Scene) — исключение с рамкой.** Активная мини-игра тикает локально (patience гостей, spawn) ради отзывчивости, но:
- смена стартует мутацией `shift.start` → сервер фиксирует `startedAt` и выдаёт сид спавна;
- очередь гостей **детерминирована** от сида (клиент и сервер генерят одинаковую) — клиент не может «нафармить» гостей;
- итог смены — одна мутация `shift.end({served, tips})`, сервер пересчитывает по сиду и своим часам, принимает свой расчёт как истину (клиентский — только для показа в реальном времени).

**Rollover и фазы недели.** `clock.weekPhase` и якоря (`rolloverAt`, `fairWindow`) приходят снапшотом и обновляются Realtime-сообщением `week:rollover`. Клиент не вычисляет фазу от локального времени сам (таймзоны, `01-core-loop`) — берёт серверную, а `serverNow()` использует лишь для плавного отсчёта «до ярмарки 2д 4ч».

### 3.7 Система заглушек-ассетов (примитивы + палитра)

Финальные low-poly GLB приходят из Blender-пайплайна (`tools/08_export.py`, как в прототипе). До тех пор — и как fallback при отсутствии файла — всё рисуется **примитивами** из общей палитры. Наследуется эстетика прототипа (CLAUDE.md): `MeshLambertMaterial`, плоские цвета из `palette.json`, ноль PBR/текстур/envMap — «как Blender solid viewport».

**Реестр `assetRegistry` (`scene/assets/registry.ts`).** Единый мап `assetKey → { placeholder, final }`:

```ts
// пример записи
'bld_diner': {
  placeholder: { shape: 'box', size: [3, 2, 2.5], color: 'diner_red', accent: 'chrome' },
  final: '/assets/props/diner.glb',   // пусто/отсутствует → рисуем placeholder
}
```

- Компонент `<Prop assetKey="bld_diner" />` смотрит реестр: если `final` есть и файл грузится — `useGLTF`; иначе — генерит примитив по `placeholder`-описанию. **Компонент не знает** про конкретный ассет — только про ключ.
- Замена заглушки на финал = добавить путь в поле `final` **одной строки** реестра. Ноль правок в сценах/компонентах.
- Ключи ассетов = ключи канона (`bld_*`, `st_*`, `tp_*`, `toy_*`, `cos_*`) плюс примитивы окружения (`env_tree`, `env_bush`, `env_road`).

**Конвенция имён заглушек:** `ph_<domain>_<name>` в отладке (`ph_bld_diner`), но в реестре ключ — каноничный (`bld_diner`). Примитивы-формы: `box | cylinder | cone | sphere | plane | group[...]` (композит из нескольких форм для сложных пропсов, напр. дайнер = box-корпус + box-навес + cylinder-вывеска).

**Палитра (`public/palette.json`).** Именованные цвета (не hex в коде): `diner_red`, `chrome`, `neon_pink`, `neon_teal`, `crop_tomato`, `crop_greens`, `field_soil`, `road_asphalt`, `sky_dusk` и т.д. Наследует и расширяет палитру прототипа. Меняем настроение сцены — правим `palette.json`, не компоненты.

**Реестр `placeholder→final` (таблица прогресса ассетов, §4.4)** ведётся здесь как единый список: что уже финальное, что ещё примитив. Это же — чеклист арт-пайплайна для спеки `18-art` (нейминг-кандидат).

### 3.8 Роутинг и дебаг (`?screen=`)

**Прод-навигация — не URL, а стор.** `scene.active` определяет активную сцену; кнопки/календарь вызывают `scene.goto(x)`. URL в проде не меняется (SPA, одна точка входа) — это упрощает деплой и убирает проблему deep-link в незагруженное состояние.

**Дебаг-оверрайды (только dev/Playwright), читаются из query при старте:**

| Параметр | Пример | Действие |
|---|---|---|
| `?screen=` | `?screen=fair` | Форс-открыть сцену (минуя условия календаря) |
| `?panel=` | `?panel=demand_board` | Сразу открыть UI-панель (`ui_*` канон §3.12) |
| `?seed=` | `?seed=42` | Детерминированный сид (спавн смены, мок-демоборд) |
| `?town=` / `?street=` | `?town=demo` | Подключиться к мок/демо-городу |
| `?net=` | `?net=offline` | Форс-оффлайн (тест очереди без dev-tools) |
| `?perf=` | `?perf=lite` / `?perf=hud` | Форс Lite mode / показать `r3f-perf` HUD |
| `?clock=` | `?clock=sat_fair` | Подставить фазу недели (тест окон без ожидания) |

Дебаг-параметры **игнорируются в проде** (гейт `import.meta.env.DEV || e2e-flag`), чтобы игрок не открывал ярмарку в понедельник через URL. Playwright-смоуки (§3.10) гоняют экраны именно через эти параметры + мок-Supabase.

### 3.9 Перф-бюджеты и рендер-стратегия

**Целевая машина (canon-адаптация «средний ноутбук»):** Intel Iris Xe / Apple M1 / интегрированная Vega, 1080p, Chrome/Safari актуальной версии. **Цель — стабильные 60 fps** в `farm` и `fair`; **≥ 40 fps** в `town` (плотная сцена) с авто-даунгрейдом до Lite при просадке.

**Бюджеты на кадр (гипотезы, финализируются профилировкой):**

| Метрика | Farm | Town | Fair | Shift | Как держим |
|---|---|---|---|---|---|
| Draw calls | ≤ 120 | ≤ 150 | ≤ 130 | ≤ 80 | инстансинг, merge статики, атлас материалов |
| Треугольники в кадре | ≤ 250k | ≤ 350k | ≤ 300k | ≤ 150k | low-poly, LOD, frustum-culling |
| Уникальные материалы | ≤ 10 | ≤ 12 | ≤ 12 | ≤ 8 | общая палитра, `MeshLambertMaterial` |
| Текстуры | 0 (плоские цвета) | 0 | 0 | 0 | ноль PBR (наследие прототипа) |
| JS heap | ≤ 250 МБ | ≤ 400 МБ | ≤ 300 МБ | ≤ 250 МБ | размонтаж `<Canvas>` при смене сцены |
| Cold start до интерактива | ≤ 3.5 с | — | — | — | code-split по сценам, ленивый GLB |

**Техники (обязательные):**
- **Инстансинг** для всех повторяющихся пропсов: грядки-слоты, деревья/кусты, гости очереди, фермы города. Одна матрица на инстанс; **один** `AnimationMixer` на тип (не на объект) — наследие прототипа. Вариативность фаз анимации — через шейдер от мировой позиции, не отдельные миксеры.
- **Город = LOD + имперостеры.** 100–200 ферм не рисуются полными мешами: дальние — билборд-имперостер (1 quad), ближние (в фокусе стрита) — упрощённый меш. Переключение по расстоянию до камеры.
- **Code-splitting по сценам:** `React.lazy` на каждую `*Scene`; GLB грузятся лениво при монтировании сцены и выгружаются при уходе (`useGLTF.clear`).
- **`<PerfGate/>`** (`scene/common`) — раз в N кадров читает fps (через `drei` `PerformanceMonitor`); при устойчивой просадке < 45 fps 2 с → `ui.liteMode = true`.
- **Lite mode:** отключить тени, урезать плотность травы/декора, снизить `dpr` до 1, город — только имперостеры, отключить пост-эффекты (если появятся). Игровая логика не меняется.
- **`dpr` адаптивный:** `[1, 1.5]` через `<Canvas dpr={...}>` + `PerformanceMonitor`; ретина не жжёт fps впустую.
- **Профилировка:** `r3f-perf` HUD в dev (уже в прототипе), включается `?perf=hud`. CI-перф-гейт — гипотеза (§8).

### 3.10 Тестовая стратегия

Наследует контракт прототипа: **`pnpm test` и `pnpm build` зелёные до коммита** (CLAUDE.md).

**Уровень 1 — vitest юниты (node, без браузера).** Обязательны для всего `game/`:
- `game/econ/*` — эконом-формулы из `14-economy` (спрос `D_cat`, перенасыщение `S_sat`, прайсинг Dimes `ceil(0.41·t^0.53)`, farm value). **Гейт: ≥ 90% покрытия строк `game/econ/`** — это деньги игроков, ошибка недопустима.
- `game/store/*` — slice-редьюсеры как чистые функции: посадка/полив/сбор, оптимистичная дельта + откат (inverse-патч), дренаж очереди (мок).
- `game/clock/*` — расчёт `serverOffset`, `serverNow`, «готово?» на границах, поведение при переводе часов (не должно ускорять).
- Конфиг — как в прототипе: `environment: 'node'`, `globals: true`. Ноль импортов three в тестируемых модулях (граница гарантирует запускаемость).

**Уровень 2 — Playwright смоуки (браузер, мок-сервер).** Не полноценный e2e с живым Supabase, а **смоук**: каждый ключевой экран открывается, рендерится, реагирует.
- Supabase перехватывается моком (route-interception `page.route('**/rest/v1/**')` + фикстуры-снапшоты в `src/test/fixtures/`); RPC-мутации отвечают детерминированно.
- Прогоны через `?screen=` + `?seed=` + `?town=demo` + `?net=`:
  1. `?screen=farm` — грядки кликабельны, посадка показывает семя (оптимистично), тост подтверждения.
  2. `?screen=fair&clock=sat_fair` — прилавок открыт, выкладка стока.
  3. `?screen=shift&seed=42` — смена стартует, детерминированная очередь, подача, итог.
  4. `?screen=town&town=demo` — город рисуется, клик по стриту фокусирует.
  5. `?net=offline` — мутации ставятся в очередь, плашка «оффлайн»; возврат online → «синхронизировано», очередь пуста.
- Смоук проверяет: сцена смонтировалась (canvas есть), нет console-error, ключевой DOM-элемент HUD присутствует, базовый клик даёт визуальный отклик. **Не** проверяет попиксельно (это дизайн-ревью).

**CI-гейт:** `pnpm --filter sunnyside test && pnpm --filter sunnyside build && pnpm --filter sunnyside exec playwright test` — все зелёные до мержа. Playwright гоняется на headless-Chromium.

| **Оптимистичная дельта валют** | HUD-баланс | серверное число + бледная `+N`/`−N` до подтверждения; схлоп при `confirmed` |
| **Тост-конфликт** | нижний-левый | `That one didn't take — try again` при `rollback`; никогда не красный (пиллар P3) |
| **Таймеры «готово»** | над грядкой/станком/прилавком | отсчёт от `serverNow()`; по истечении — бейдж `Ready! (Готово!)`, забор кликом |
| **Кнопки смены сцены** | HUD | `To Town / To the Fair`; ярмарка активна только в окне `sat_fair` (иначе disabled с подсказкой «Ярмарка в субботу») |
| **Lite-mode индикатор** | настройки | тумблер + авто-статус; «Мы упростили графику ради плавности» |
| **Debug-оверлей** | dev only | активные `?`-параметры, fps (r3f-perf), длина очереди, `serverOffset`, фаза недели |
| **Snapshot-бейдж** | town/fair офлайн | `Snapshot — reconnecting` поверх устаревших данных |

Все строки — EN + RU через `ui.locale` (канон §5, словарь локализации). Дизайн и вёрстка экранов — спека `19-ui-ux` (нейминг-кандидат).

## 7. Edge cases

| # | Ситуация | Поведение клиента |
|---|---|---|
| C1 | Обрыв связи в момент RPC | Мутация остаётся оптимистичной, вернулась в `queued`; при online — ретрай по `clientMutationId` (без дубля эффекта) |
| C2 | Игрок закрыл вкладку с непустой очередью | Очередь в IndexedDB переживает; при следующем входе (`main.tsx` шаг 6) доигрывается до подписок на сцену |
| C3 | Часы ноутбука сбиты/переведены вперёд | Всё считается от `serverNow()`; таймеры не ускоряются, забор ресурса валидирует сервер (§3.6) |
| C4 | `serverOffset` не замерился (RPC упал на старте) | Работаем в оптимистик-офлайне от локальных часов **только для отображения**; логику готовности блокируем до первого успешного `get_server_time` (бейдж «синхронизируем время») |
| C5 | Конфликт: слот занят на сервере (сосед-гонка невозможна — слот личный, но повтор из очереди) | Идемпотентность вернёт закэшированный `ok`; если реально конфликт — `rollback` + тост, слот освобождается |
| C6 | Realtime отвалился, REST жив | Периодический poll-фолбэк town/fair снапшотов (гипотеза 30 с), бейдж `Snapshot`; при возврате Realtime — отписка от poll |
| C7 | Слабое железо, стабильно < 30 fps даже в Lite | Остаёмся в Lite, `dpr=1`, без теней; предлагаем свернуть город к списку стритов (2D-фолбэк town, гипотеза) |
| C8 | Отсутствует GLB-файл (404) | `assetRegistry` тихо падает на `placeholder`-примитив; ошибка в dev-консоль, игрок видит заглушку, не пустоту |
| C9 | Прод-игрок подставил `?screen=fair` в понедельник | Дебаг-параметры отключены вне dev/e2e (§3.8) — игнор, открывается `farm` |
| C10 | Смена завершилась, но `shift.end` не дошёл (офлайн) | Итог в очереди; сервер пересчитает по сиду/часам при доставке — начисление корректное, не по клиентскому числу |
| C11 | Персист-кэш устарел (версия схемы клиента сменилась) | Версионированный ключ persist; при несовпадении — сбросить кэш-снапшоты (не очередь!), ре-гидрация с сервера |
| C12 | Двойной клик по «собрать» | Дебаунс + `clientMutationId` идемпотентность: второй клик не создаёт вторую мутацию |
| C13 | Игрок открыл игру в двух вкладках | Каждая — свой оптимистик; сервер-истина сходится через Realtime; очередь в IndexedDB общая — дренаж координируется через `BroadcastChannel` (гипотеза), лидер-вкладка флашит |

## 8. Открытые вопросы

1. **Номер спеки бэкенда.** Сетевой слой (§3.5) блокирующе зависит от контрактов Edge Functions, но спеки бэкенда пока нет. Предложен `docs/specs/20-backend.md` (нейминг-кандидат) — согласовать номер и создать; до этого RPC-сигнатуры в §3.5/§4 — гипотезы.
2. **Номер спеки арта.** Реестр `placeholder→final` (§3.7) ждёт спеку GLB-пайплайна; предложен `18-art.md` (нейминг-кандидат). Подтвердить, что арт наследует Blender-пайплайн прототипа (`tools/`) или заводит свой.
3. **Расхождение нумерации экономики.** Канон и сиблинги ссылаются на `12-economy.md`, фактически файл — `14-economy.md` (см. его §8). Клиент ссылается на `14-economy`; финализировать одно имя во всех спеках.
4. **Монорепо-инструмент.** `pnpm --filter sunnyside` подразумевает workspace. Заводить `pnpm-workspace.yaml` (прототип + клиент как пакеты) или держать `sunnyside/` как вложенный независимый проект с отдельным lock? Влияет на CI-команды (§3.10).
5. **`clientMutationId` окно 24 ч.** Достаточно ли для игрока, вернувшегося из офлайна через > 24 ч с недренированной очередью? Возможно нужен «мягкий отказ» устаревших мутаций (гипотеза: старше 48 ч — дропать с уведомлением). Согласовать со спекой бэкенда.
6. **Координация мультивкладок (C13).** `BroadcastChannel`-лидер для дренажа очереди — гипотеза. Нужен ли вообще, или сервер-идемпотентность делает гонку вкладок безопасной без координации?
7. **2D-фолбэк города (C7).** Для очень слабого железа — свернуть `TownScene` к списку стритов? Или гарантировать имперостерами 3D всем? Требует плейтеста на нижнем пороге железа.
8. **CI-перф-гейт.** Автоматически валить сборку при регрессе fps/draw calls — как измерять в headless-CI детерминированно (нет реального GPU)? Возможно только ручная профилировка + бюджет-линт на счётчики (гипотеза).
9. **Poll-фолбэк частота (C6).** 30 с для town/fair при мёртвом Realtime — гипотеза; согласовать с нагрузкой на Edge/REST в спеке бэкенда.
10. **Auth-модель.** Анонимная сессия vs magic-link на входе — влияет на `session`-слайс и гидрацию. Решается в спеке бэкенда; клиент готов к обоим.
11. **Локаль по умолчанию.** `ui.locale` стартует по `navigator.language` или всегда EN с переключателем? Канон §5 требует двуязычие — уточнить дефолт и полноту RU-перевода строк сетевого слоя.

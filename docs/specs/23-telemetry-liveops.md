## 1. TL;DR

Sunnyside — асинхронный сервер-центричный лайвсервис: игровая неделя = реальная неделя, экономика и ивенты живут на Supabase (Postgres + Realtime + Edge Functions). Телеметрия должна отвечать на три вопроса без единой строки нового кода на клиенте: (1) доходит ли новичок до первой ярмарки и первого спенда, (2) здоров ли ритм недели по городам (Пн–Вс), (3) что чинить сегодня, если город/стрит просел. Стек аналитики: клиентский трекер (батч-отправка событий в Edge Function `ingest-events` → таблица `analytics_events` в Postgres, партиционированная по неделе) + вьюхи/материализованные представления для воронок и KPI + дашборд-приложение (админка, отдельный React-роут, доступ по роли `ops`/`support`). Remote config — таблица `game_config` (JSON per key), читаемая клиентом при старте сессии и кешируемая 15 мин; правки — только через админку с аудит-логом, без деплоя клиента. Модерация — префильтр никнеймов/чата на клиенте (словарь + Edge Function проверка) + пост-модерация через очередь репортов в админке. GDPR — минимальный набор (согласие на аналитику, экспорт/удаление по запросу, TTL сырых событий 400 дней, PII не пишется в события).

Ключевые числа (гипотеза, если не из канона):

| Параметр | Значение |
|---|---|
| Батч отправки клиентских событий | каждые `10 сек` или `20 событий` (что раньше), + flush при `visibilitychange: hidden` |
| Ретеншн сырых событий (`analytics_events`) | `400 дней` (гипотеза), затем архив в холодное хранилище (агрегаты остаются вечно) |
| Обновление remote config без деплоя | вступает в силу ≤ `15 мин` (TTL кеша клиента) |
| SLA на разбан/компенсацию через админку | ≤ `24 ч` с момента репорта (гипотеза) |
| A/B unit | **Город** (Town), не игрок — из-за серверной природы недели |
| Согласие на аналитику | opt-out модель для функциональной телеметрии (без которой сервис не работает), opt-in для маркетинговой атрибуции |

## 2. Player Experience (как это ощущается)

Игрок никогда не видит телеметрию напрямую — но чувствует её последствия:

- **Новичок**, застрявший на шаге «посади первую грядку» дольше 90 сек, через 2 мин получает мягкую подсказку-тултип от `npc_whittaker` (Старик Уиттакер) — эта подсказка триггерится по событию `ftue_step_stalled`, которое пишет клиент, но решение показать/не показать — из `game_config` (можно выключить подсказку A/B-тестом без деплоя).
- **Стрит**, где ярмарочная явка упала ниже 40% три недели подряд, автоматически подсвечивается в админке support-команде флагом `street_attendance_risk` (лидирующий индикатор на уровне Стрита, метрика — явка на ярмарку, НЕ DAU) — если игрок напишет тикет «у нас на стрите тихо», агент видит контекст сразу, а не расследует с нуля. Само решение о Town Merge принимается на уровне Town по DAU (DECISIONS-B K5, см. §3.7).
- **Игрок, ливнувший на FTUE-шаге 4/7**, попадает в воронку `ftue_funnel` — и если таких много на одном шаге, дизайнер узнаёт это на дашборде в течение суток, не через ревью в сторе через месяц.
- **Кит**, купивший бандл, не видит ничего особенного — но событие `purchase` с `sku`/`price_usd`/`context` даёт LiveOps понять, какие бандлы конвертят, без нужды спрашивать игроков.
- **Игрок с оскорбительным ником** блокируется превентивно при вводе (клиентский словарь + серверная Edge Function проверка) — не доходит до чужих экранов вовсе, никакого «увидел — пожаловался — жду неделю».
- **Жалоба на чат** обрабатывается модератором из очереди в админке за то же SLA, что и баг-репорт — модерация не воспринимается игроками как «отдельный, забытый ящик».

## 3. Механики — исчерпывающе

### 3.1 Архитектура пайплайна телеметрии

```
Клиент (React) → EventQueue (zustand slice, in-memory + IndexedDB buffer)
   → батч POST /functions/v1/ingest-events (Edge Function, каждые 10с/20 событий/hidden)
      → валидация схемы (zod) + rate-limit (макс 200 событий/мин/устройство)
         → INSERT analytics_events (Postgres, партиция по ISO-неделе)
            → триггер: обновление realtime-агрегатов (town_daily_stats, funnel_progress)
            → раз в час: cron Edge Function пересчитывает материализованные вьюхи KPI
```

- Каждое событие: `event_name`, `ts` (client + server), `user_id`, `session_id`, `town_id`, `street_id`, `app_version`, `platform`, `params` (jsonb), `server_week_id` (ISO-неделя игрового календаря, canon §2.3).
- Клиент не блокируется на отправке — при офлайне события копятся в IndexedDB (кап `500` событий, старые дропаются с event `telemetry_buffer_overflow`).
- Дедупликация: `event_id` (uuid, клиентский) — уникальный индекс в БД, повторная отправка при retry не создаёт дубль.
- Никаких PII в `params`: email/имя реального игрока не пишутся; только `user_id` (внутренний uuid) и игровые сущности (ключи из канона: `staff_bruno`, `ui_fair_stall` и т.д.).

### 3.2 Полный реестр событий (57 событий)

Группы: `session`, `ftue`, `farm`, `craft`, `economy`, `fair`, `event` (server event), `social`, `progression`, `monetization`, `retention`, `moderation`, `system`.

См. таблицу 4.1 для полного списка с параметрами (сквозная нумерация 1–57).

### 3.3 Воронки (funnels)

**Воронка A — FTUE (Onboarding).** Шаги 1–7 из `18-onboarding.md`, каждый шаг = событие `ftue_step_completed{step, step_key}`. Метрика: % прошедших шаг N при старте N-1, время на шаг (медиана/p90). Алерт: если % прохождения шага < `70%` за скользящие 24 ч — пуш в Slack ops-канал (гипотеза).

**Воронка B — первая ярмарка.** `signup` → `ftue_completed` → `first_plot_harvested` → `first_dish_cooked` → `fair_stall_opened_first_time` → `fair_first_sale`. Целевой % от `signup` до `fair_first_sale` за первую календарную субботу игрока: `≥ 35%` (гипотеза, целевой KPI).

**Воронка C — первый спенд.** `signup` → `shop_viewed` → `purchase_initiated` → `purchase_completed`. Средний срок до первого спенда (для конвертящих): `D7` (гипотеза). Точки утечки инструментируются отдельно: `purchase_initiated` без `purchase_completed` за `10 мин` → событие `purchase_abandoned{reason}`.

**Воронка D — участие в кооп-заказе (Co-op Orders).** `coop_order_viewed` → `coop_order_joined` → `coop_order_contributed` → `coop_order_completed`. Целевой % от `joined` до `completed`: `≥ 60%`.

**Воронка E — участие в серверном ивенте.** `event_announced_seen` → `event_contrib_first` → `event_milestone_reached` (любой) → `event_final_reward_claimed`. См. `10-server-event.md`.

### 3.4 KPI-реестр

Считается по когортам `signup_week`, разрез по `town_id`, `platform`, `acquisition_source` (если есть). Источник — материализованные вьюхи, пересчёт раз в час (некритичные) и раз в сутки (когортные).

- **Ретеншн**: D1, D3, D7, D14, D30, D60 (return-визит в игру, событие `session_start` в соответствующий день после `signup`).
- **Вовлечённость в ритуал недели**: `% игроков стрита, открывших Demand Board в Пн`, `% игроков, сделавших ≥1 крафт во вторник-пятницу`, `% явки на ярмарку в субботнее окно (36 ч)`, `% участия в серверном ивенте (внёс ≥1 вклад в Общий котёл)`.
- **Монетизация**: ARPDAU, ARPPU, конверсия в платящих (D7/D30), LTV D30/D90 (модельная экстраполяция), % дохода от `dimes`-покупок vs `bundles`.
- **Экономика здоровья**: средний баланс `bucks`/`dimes` по когорте, инфляция цен на Demand Board (дрейф спроса), % «мёртвых» складов (переполнение → E3 канона).
- **Соц-метрики**: средний размер активного стрита (DAU/стрит), % Street Potluck участия по субботам, число Town Merge/Street Caravan событий в неделю.
- **Стабильность/качество**: crash-free sessions %, p95 задержки `ingest-events`, % событий, потерянных из-за офлайн-переполнения буфера.

### 3.5 Remote config — что конфигурируемо без деплоя

Таблица `game_config` (ключ → jsonb значение, версия, `updated_by`, `updated_at`, опционально `town_scope` для A/B). Категории:

| Категория | Примеры ключей | Кто правит |
|---|---|---|
| Экономика | `price.dish.T1..T5`, `price_multiplier_by_tier`, `craft_time_seconds.*`, `storage_cap.silo/icehouse` | LiveOps + баланс |
| Спрос/ярмарка | `demand_board.volatility_pct` (±15–30%, canon §2.3), `fair_window_hours` (36, canon), `fair_stall_passive_rate` | LiveOps |
| Ивенты | `event_schedule` (ротация `ev_*` из канона §3.5), `appetite_meter.milestones` (25/50/75/100%), `event_final_hour_utc` (**20:00 UTC — решено, DECISIONS-B K7**, фиксировано круглогодично) | LiveOps |
| Монетизация | `bundle_catalog`, `dimes_price_tiers`, `daily_boost_cap`, `prize_machine_pity_counter_rare` (гарант ≤10), `prize_machine_pity_counter_chase` (гарант ≤40) — открытый двухступенчатый pity, дроп 68/24/6.5/1.5% (мастер — `15-monetization.md §3.3`, DECISIONS-B K2) | Монетизация-лид |
| Ретеншн | `streak_insurance_cost_tickets`, `vacation_mode_max_days`, `winback_wave_schedule` | LiveOps |
| Feature flags | `feature.town_projects_enabled`, `feature.expeditions_v2`, `ab_test.<key>.variant_weights` | Продукт/инженер |
| Модерация | `chat_filter_wordlist_version`, `nickname_filter_strictness`, `auto_mute_threshold_reports` | Trust & Safety |

Правила:
- Любое значение — `default` + опциональные override по `town_id`/`street_id`/`ab_bucket`.
- Изменение пишется в `game_config_audit` (кто/когда/старое/новое значение) — обязательный аудит-лог (см. §3.7).
- Клиент кеширует конфиг `15 мин` (TTL), досрочный сброс кеша — через Realtime-канал `config_invalidate` (мгновенный сброс критичных ключей, напр. экстренный выключатель бандла).
- Экономические ключи меняются не чаще `1 раз/неделю` в понедельник в окно `mon_plan` (canon), чтобы не ломать баланс посреди недели — за исключением hotfix-паники (см. §7).

### 3.6 A/B-фреймворк (на уровне городов)

Причина unit = **Town**, а не Player: игровая неделя и ивенты — серверные шкалы, общие для всего города (Appetite Meter, Demand Board), эксперимент на уровне игрока сломал бы общие механики (сосед в другом варианте видел бы другую экономику одного и того же города).

- Юнит эксперимента: `town_id`. Назначение варианта — при создании города (детерминированный хэш `town_id + experiment_key` → бакет), фиксируется в `town_experiments` (town_id, experiment_key, variant, assigned_at).
- Минимальный размер выборки: `20 городов на вариант` (гипотеза, из-за 100–200 игроков/город и ≈40–60 DAU — нужно больше городов, чем в player-level экспериментах, для той же статистической мощности).
- Длительность теста: **не менее 3 игровых недель** (после недели «приладки» новых городов) — потому что ключевые метрики (явка на ярмарку, участие в ивенте) имеют недельную сезонность.
- Гардрейлы теста: экономические A/B (цены, множители) не могут давать разницу дохода игроков > `±20%` между вариантами (этический гардрейл — не создавать заведомо «городов второго сорта»); монетизационные A/B не трогают гарантии D11 канона (прямая продажа силы запрещена в любом варианте).
- Инструментарий: события помечаются `ab_bucket` в `params` автоматически на сервере (не клиентом — чтобы нельзя было подделать/обойти); дашборд A/B — отдельная вкладка админки, значимость считается Bayesian (вероятность что B > A ≥ `95%`) из-за медленного накопления данных по городам.
- Остановка теста: ранняя остановка разрешена только при триггере гардрейла (см. выше) или явном проседании retention D7 варианта > `5 п.п.` относительно контроля — иначе тест идёт до конца заявленного срока.

### 3.7 Админ-инструменты (дашборд, компенсации, баны)

Отдельное защищённое SPA (роут `/ops`, RLS на Supabase по роли `ops_admin`/`ops_support`/`trust_safety`), тянет те же таблицы read-only + пишет команды через Edge Functions с аудитом.

**Дашборд города** (`/ops/towns/:town_id`):
- Сводка: DAU/недельные активные, распределение по стритам, явка на ярмарку по субботам (график), % участия в текущем ивенте, health-score (композит из воронок C/D/E).
- Список стритов с флагами риска (`явка < 40% три недели` → `risk: street_attendance_risk`, лидирующий индикатор). Это НЕ тот же сигнал, что триггерит Town Merge: решение о мердже принимается по агрегату **Town DAU** (DECISIONS-B K5 — мягкий флаг при 7-дневном среднем Town DAU `<40` (баннер+стимулы), активное предложение-мердж при Town DAU `<25`). Стрит-флаг явки агрегируется вверх в Town DAU через `town_daily_stats` и служит ранним предупреждением support-команде, финальный порог решения — Town-уровень (см. `12-migration.md §3.3.1`, `11-town.md §7 T6`).
- Таймлайн событий города (Town Merge, Street Caravan, Town Project завершения).
- Кнопка «форс-пересчёт Demand Board» и «продлить окно ярмарки на N часов» (аварийный клапан, пишется в аудит с обязательным полем `reason`).

**Компенсации** (`/ops/compensations`): форма — `user_id` или `town_id`, тип (`bucks`/`dimes`/`tickets`/`item`/`streak_restore`), сумма, причина (свободный текст, обязателен), привязка к тикету поддержки. Все компенсации — событие `admin_compensation_granted{admin_id, target, currency, amount, reason, ticket_ref}`. Лимиты: одиночная выдача `dimes` ≤ `500` без второй подписи (four-eyes review для больших сумм, гипотеза-порог); дневной кап на одного `ops_support`-агента — `5000 dimes`-эквивалента.

**Баны/муты** (`/ops/moderation`): очередь репортов (ник/чат), карточка нарушения (контекст, скриншот-снапшот сообщения, история репортов на пользователя), действия: `warn` / `mute_24h` / `mute_7d` / `rename_force` / `ban_permanent`. Бан — не блокирует прогресс фермы (её видно, но управлять нельзя во время бана) — сохраняет дух P3 (никакого разрушения чужого прогресса). Апелляция — отдельная форма, SLA рассмотрения `≤ 48 ч` (гипотеза).

**Аудит-лог** (`/ops/audit`): неизменяемая таблица (append-only, `admin_id`, `action`, `target`, `before`, `after`, `reason`, `ts`) — обязательна для всех write-действий из админки (config-правки, компенсации, баны, форс-пересчёты).

### 3.8 Модерация (имена, чат)

- **Никнеймы**: клиентский префильтр (локальный словарь стоп-слов, обновляется через `game_config.chat_filter_wordlist_version`) при вводе + серверная Edge Function `validate_nickname` (проверка при регистрации/смене имени) — блокирует явный мат/оскорбления/попытки выдать себя за NPC/staff (`Sheriff`, `Mayor` и т.п. — зарезервированные подстроки). При срабатывании — предложение 3 альтернатив (авто-сгенерированных в духе 50-х: `Pixie Lane`, `Rusty Wagon`...).
- **Чат** (Street/Town уровень): чат — нормативная фича MVP (мастер-док соцсистем `11-town.md §3.9`, DECISIONS-B 11-O1: `11-town.md` — единый мастер соцсистем, чат существует, не опционален). Фильтр по словарю + rate-limit (`10 сообщений/мин`, гипотеза) + репорт-кнопка на каждом сообщении. Репорт создаёт запись в очередь модерации с приложенным контекстом (последние 5 сообщений треда).
- **Авто-мут**: `auto_mute_threshold_reports` (remote config) — базовый порог `≥ 3` уникальных репортов на одно сообщение за `1 ч` — авто-скрытие сообщения + временный мут до ручной проверки (защита от вала, а не финальное наказание). Порог **взвешивается**, а не считается по сырому количеству репортов, против report-brigading (пилларс P3 — не давать групповому давлению банить невиновного): (1) репорт от аккаунта без истории взаимодействия с автором сообщения (не в одном стрите/городе, не в общем треде) весит `0.5×`; (2) репорты с низким trust-score (недавно созданный аккаунт, `account_age_days < 3`, либо сам ранее банился/мутился) весят `0.25×`; (3) дополнительный гейт разнообразия устройств — авто-мут не срабатывает, если ≥ 60% репортов пришли с похожего device-fingerprint кластера (признак координированной атаки с мульти-аккаунтов). Взвешенная сумма ≥ 3 (эквивалент трёх «полновесных» независимых репортов) — триггер авто-скрытия; порог и веса конфигурируются через `game_config.auto_mute_threshold_reports`/`auto_mute_weight_*` (Trust & Safety).
- **Тон-гейт**: любое авто-действие модерации формулируется мягко на клиенте («Сообщение скрыто на проверку» — не «Вы забанены за токсичность») — соответствует пилларсу P3, тон не карательный.
- **Апелляции и false positive**: игрок может оспорить мут/бан из настроек профиля — попадает в ту же очередь `/ops/moderation` с пометкой `appeal`.

### 3.9 GDPR / приватность (минимум)

- **Согласие**: при первом запуске — баннер с двумя переключателями: (1) «Функциональная аналитика» (включена по умолчанию, opt-out — без неё часть liveops-функций типа персонализированных подсказок деградирует, но игра работает), (2) «Маркетинговая атрибуция» (opt-in, выключено по умолчанию, для UA-партнёров/рекламных SDK, если такие появятся v1.x).
- **PII-минимизация**: `analytics_events.params` не содержит email, реальное имя, платёжные данные — только `user_id` (внутренний uuid, не совпадает с auth id напрямую — маппинг только через защищённую таблицу `user_identity_map`, доступную единицам ролей `ops_admin`).
- **Право на экспорт**: запрос через саппорт (или самостоятельно из настроек, v1.x) → Edge Function собирает все данные пользователя (профиль, ферма, транзакции, события за последние 400 дней) в JSON, отдаёт ссылку со сроком жизни `72 ч`.
- **Право на удаление**: soft-delete профиля (`deleted_at`) сразу, hard-delete PII + анонимизация событий (`user_id` → `deleted_user_hash`) в течение `30 дней` (гипотеза, стандартный грейс-период на случай отмены запроса).
- **Дети/возрастные ограничения**: без сбора данных, требующих верификации возраста за пределами стандартной формы сторов (App Store/Google Play возрастной рейтинг); не таргетируется явно на детей до 13 (COPPA-ориентир) — данные о поведении несовершеннолетних не продаются третьим лицам (нет продажи данных вообще, только внутренняя аналитика и опциональная UA-атрибуция при explicit opt-in).
- **Хранение**: сырые события — `400 дней` (гипотеза), затем удаляются или агрегируются без `user_id`; агрегаты (KPI-вьюхи) хранятся бессрочно (уже не PII).
- **Third-party SDK**: любой сторонний аналитический/рекламный SDK подключается только за `feature.marketing_sdk_enabled` флагом remote config и только после явного opt-in — по умолчанию выключен на старте MVP (весь трекинг MVP — first-party, через собственный `ingest-events`).

### 4.5 Лимиты компенсаций в админке

| Роль | Лимит на выдачу без second-review | Дневной кап |
|---|---|---|
| `ops_support` | `dimes ≤ 500` / `bucks ≤ 5,000` / `tickets ≤ 20` | `5,000 dimes`-экв/день |
| `ops_admin` | `dimes ≤ 2,000` / без ограничений bucks/tickets | `20,000 dimes`-экв/день |
| Любая сумма выше | требует подтверждения второго `ops_admin` (four-eyes) | — |

**Формула пересчёта в `dimes`-эквивалент (для дневного капа).** В игровой экономике курс `◉→$` отсутствует и не вводится (гардрейл анти-P2W D11, `14-economy.md §3.2`) — поэтому ниже НЕ игровой обменный курс, а **внутренняя учётная константа только для ops-инструмента** (капы админ-выдач, не влияет на баланс игры и нигде не показывается игроку):

```
dimes_equivalent = dimes_amount
                  + bucks_amount / 100        // 100 $ ≈ 1 ◉-экв (учётная константа ops-капа)
                  + tickets_amount × 2        // 1 🎟 ≈ 2 ◉-экв (учётная константа ops-капа)
```

- Дневной кап `ops_support` (`5,000 dimes-экв`) и `ops_admin` (`20,000 dimes-экв`) считается по сумме `dimes_equivalent` всех компенсаций агента за календарные сутки (UTC), не по одной валюте.
- Константы `100 $/◉` и `2 ◉/🎟` калибруются по факту типичных сумм компенсаций (не по игровой ценности валют) и правятся через `game_config.ops_dimes_equivalent_rates` (Trust & Safety/LiveOps-лид), с записью в `game_config_audit`.
- Item-компенсации (`item`, `streak_restore`) не входят в `dimes_equivalent` — считаются штучно (см. §3.7), но подлежат отдельному внутреннему логированию частоты на агента.

## 7. Edge cases

| # | Ситуация | Поведение |
|---|---|---|
| T1 | Клиент офлайн долго (Vacation Mode, `mech_gone_fishin`) | События не шлются; при возврате — буфер IndexedDB (до 500 событий) догружается батчем с корректными исходными `ts`; события старше `400 дней` буфера отбрасываются молча |
| T2 | Всплеск нагрузки в момент открытия ярмарки (все города одновременно, окно 36ч старт) | `ingest-events` — асинхронная очередь с backpressure (429 + client backoff), не блокирует геймплей — потеря части `heartbeat`-событий допустима, критичные (`purchase_completed`, `fair_result`) — с retry до 3 раз |
| T3 | Remote config откатывается посреди недели из-за бага баланса | Hotfix-путь вне обычного расписания (обычно правки экономики — только Пн mon_plan): экстренный `config_invalidate` через Realtime, немедленный сброс кеша всех клиентов, обязательная запись в аудит с `reason: hotfix` |
| T4 | A/B тест показывает статистически значимый вред retention одному варианту раньше 3 недель | Досрочная остановка разрешена по гардрейлу §3.6 — все города варианта переводятся на контрольный конфиг, `game_config_audit` фиксирует причину |
| T5 | Игрок оспаривает бан (false positive от auto-mute) | Апелляция в очередь `/ops/moderation` с пометкой `appeal`, приоритет выше обычных репортов, SLA `≤ 48 ч`; до решения — статус `muted`, не `banned` (мягче) |
| T6 | Компенсация выдана по ошибке (опечатка в сумме) | Нет отмены транзакции валюты пост-фактум (просто) — вместо этого встречная компенсация-коррекция с явной причиной `correction_of:<original_ticket_ref>`, обе записи видны в аудите |
| T7 | Запрос на удаление данных от игрока с активным незавершённым Co-op Order/событием | Soft-delete немедленно (аккаунт недоступен для входа), но вклад в общие шкалы (Appetite Meter, кооп-заказ) не откатывается — только PII анонимизируется; продукт вклада игрока в общий котёл остаётся (не наказывает других участников, P3) |
| T8 | Дубликат события из-за retry на плохой сети | `event_id` уникальный индекс — дубль игнорируется на INSERT (`ON CONFLICT DO NOTHING`) |
| T9 | Toжн — эксперимент A/B требует минимум 20 городов на вариант, но сервер ещё маленький (< 40 городов всего) | Тест не стартует до достижения минимального числа городов; вместо A/B — контролируемый последовательный rollout (canary: 5 городов → неделя наблюдения → остальные) |
| T10 | Утечка PII в `params` события по ошибке разработчика (например, случайно попал email) | Схема-валидатор (zod) на Edge Function отклоняет событие с несовпадением схемы полей; дополнительно — регэксп-скан на PII-паттерны (email/phone) как second line of defense, событие дропается с логом `schema_violation` |
| T11 | Игрок из юрисдикции с обязательным opt-in для любой аналитики (не только маркетинговой) | Remote config флаг `region_strict_privacy_mode` (по `country_code` при регистрации) переключает функциональную аналитику тоже в opt-in — деградация liveops-персонализации для этого региона допустима |

## 8. Открытые вопросы

1. **Ретеншн сырых событий 400 дней** — цифра-гипотеза; нет согласованной политики хранения данных на уровне компании/юрисдикций (может требоваться короче для EU-игроков — см. edge case T11). Согласовать с юр.отделом при появлении.
2. **A/B unit = Town** — решение этой спеки, не зафиксировано в каноне. Канон определяет соц-структуру (§2.4), но не упоминает A/B-фреймворк — предлагаю внести ссылку в канон при следующей итерации (`00-canon.md` §4, новая развилка D15: «A/B unit = Town»).
3. **Минимум 20 городов на вариант A/B** — гипотеза, требует проверки мощности теста реальными числами конверсии после первых недель MVP (когда будет фактическая дисперсия KPI по городам).
4. **Целевые проценты KPI (§4.3) и воронок (§4.2)** — все инженерные гипотезы на основе жанровых ориентиров (farm-sim/social casual), не валидированы плейтестом Sunnyside. Требуют пересмотра после первой публичной альфы — финальные цифры централизовать в `14-economy.md`/`16-retention.md` там, где они пересекаются.
5. **SLA компенсаций 24ч и апелляций 48ч** — гипотезы, не согласованы с реальным штатом support на запуске (зависит от размера команды LiveOps, которого нет в этой спеке).
6. **Second SDK / маркетинговая атрибуция** — этот документ предполагает, что на MVP их нет вообще (only first-party). Если паблишер/UA-стратегия v1.x потребует Adjust/AppsFlyer — потребуется отдельный ревью GDPR-раздела (§3.9) и, возможно, отдельная политика согласий по регионам.
7. **Four-eyes порог для компенсаций (§4.5)** — цифры `500/2000 dimes` — гипотезы без экономического якоря. Игровой курс `◉→$` в принципе не существует (D11, анти-P2W) — введённые в §4.5 константы `100 $/◉-экв`, `2 ◉-экв/🎟` являются исключительно внутренней учётной единицей ops-инструмента (не игровым курсом) и требуют калибровки по фактическим суммам support-кейсов после запуска, а не по игровой ценности валют.
